# [FEATURE]: Home Assistant Repairs Proxy Service

**Target Repo:** `srat` **Status:** 📅 Planned **Issue Link:** _TBD_

## 🎯 Objective

Create a backend repair-management service that can request, track, and resolve Home Assistant Repairs through the SRAT custom component over the existing WebSocket server→client channel. The custom component should proxy Home Assistant Repairs platform capabilities back to the backend, report repair lifecycle events over WebSocket, and support queued repair messages when the custom component is temporarily disconnected.

## 🛠️ Technical Specifications

- **Inputs:**
  - Backend repair-management requests generated by SRAT services
  - Existing `/ws` SRAT WebSocket connection and `helo` handshake state
  - Home Assistant Repairs platform semantics from `https://developers.home-assistant.io/docs/core/platform/repairs`
  - Repair lifecycle callbacks/events emitted by the custom component
  - Custom component connection status (connected/disconnected)

- **Outputs:**
  - Backend service API for creating, updating, deleting, and reconciling managed Repairs requests
  - New WebSocket message contracts for backend→custom-component repair commands
  - New inbound WebSocket events for custom-component→backend repair lifecycle updates
  - Queueing/flush behavior for repair commands when the custom component is not connected
  - Custom component proxy implementation that bridges backend repair commands to Home Assistant `issue_registry` / `RepairsFlow` behavior

- **Dependencies:**
  - Backend WebSocket transport: `backend/src/api/ws.go`, `backend/src/service/broadcaster_service.go`
  - Backend DTO/message contracts: `backend/src/dto/helo.go`, `backend/src/dto/webevent_map.go`, `backend/src/dto/webevent_type.go`
  - Backend eventing/state: `backend/src/events/`, `backend/src/dto/context.go`
  - Custom component runtime: `custom_components/srat/websocket_client.py`, `custom_components/srat/__init__.py`
  - New custom component repair proxy module(s), likely `custom_components/srat/repairs.py`
  - Home Assistant developer docs: [Repairs](https://developers.home-assistant.io/docs/core/platform/repairs)

## 🧠 Implementation Notes (Copilot Context)

- The transport direction is primarily **backend → WebSocket server→client channel → custom component → Home Assistant Repairs APIs**, with lifecycle/status events flowing back from the custom component to the backend.
- The backend already accepts inbound WebSocket JSON envelopes from the custom component (`helo`). Repair event messages should build on the same envelope-routing pattern instead of creating a second channel.
- The backend already has a broadcast-based WebSocket pipeline in `BroadcasterService`; this task should decide whether repair commands fit into that existing message path or require a dedicated repair-command sender with explicit queue semantics.
- Queueing is required when the custom component is not connected. The queue should survive temporary disconnects and be flushed in order after the next valid `helo` handshake.
- Home Assistant Repairs platform guidance indicates:
  - issues are created through `issue_registry.async_create_issue(...)`
  - fixable issues should expose a `RepairsFlow`
  - resolved/deleted issues can be removed with `issue_registry.async_delete_issue(...)`
  - ignored issues persist until deleted and recreated
- The custom component should be the only place that directly touches Home Assistant repair helpers; the Go backend should remain platform-agnostic and communicate through typed proxy messages.
- Repair commands should be idempotent where possible, using stable identifiers so reconnects do not create duplicate Home Assistant issues.
- The custom component likely needs a dedicated `repairs.py` module plus WebSocket event handlers that map backend repair operations to HA issue registry actions and repair-flow results.
- Backend state reconciliation should support events such as created, updated, ignored, fixed, dismissed, deleted, and error.
- If repair operations cannot be executed immediately because the component is disconnected, the backend should preserve intent and emit the command once connectivity is restored.
